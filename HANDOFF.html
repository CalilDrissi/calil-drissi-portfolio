1Quick start on the new machine

Prerequisites: Node.js (CI uses Node 20), Git, and Claude Code. Optional: Homebrew + Python 3.12+ for the graphify tooling.

# 1. Unzip the transfer, then enter the project (git root is v2/)
cd calil-drissi.dev/v2

# 2. Install dev deps (@playwright/test). Browsers only needed if you run tests:
npm install
npx playwright install        # optional, for tests/

# 3. .env is INCLUDED in the zip (WP_URL, WP_USER, WP_APP_PASSWORD, PEXELS_KEY) — verify:
cat .env | sed 's/=.*/=/'

# 4. Build the static site -> dist/
node build.js

# 5. Preview locally
cd dist && python3 -m http.server 8080   # then open http://localhost:8080

# 6. Deploy: just push to main (see §3). No manual deploy step.
Git remote is SSH: git@github.com:CalilDrissi/calil-drissi-portfolio.git. On the new machine either add your SSH key to GitHub, or switch to HTTPS + a Personal Access Token: 
git remote set-url origin https://github.com/CalilDrissi/calil-drissi-portfolio.git

3Architecture & deploy pipeline

build.js reads the index.html template + content/en.json / content/fr.json, fetches published blog posts from WordPress, and renders dist/: EN at /, FR at /fr/, plus blog listing + post pages. It also copies static assets and the Cloudflare Functions.

Two ways a deploy is triggered

  1. Code push: git push origin main → GitHub Actions (.github/workflows/deploy.yml, Node 20) runs node build.js → deploys dist/ to Cloudflare Pages via wrangler-action.
  2. WordPress publish: a WP publish fires the "Khalil Deploy Hook" plugin → POST /api/webhook (functions/api/webhook.js) → GitHub repository_dispatch (event wordpress_publish) → same Actions build → deploy. So new blog posts go live without a manual push.
Bulk-insert tip (from experience): when creating many posts via the CLI, temporarily gh workflow disable "Build & Deploy", insert, then re-enable + one push for a single clean deploy (avoids a deploy storm).

5Systems in detail

AI chat assistant

functions/api/chat.js → Ollama Cloud (https://ollama.com/api/chat, model gemma3:27b, Bearer OLLAMA_API_KEY — a Cloudflare secret). System prompt carries Khalil's bio; the model replies in the visitor's language and in plain text (no emoji). If the API is unreachable, the client falls back to keyword-matched canned replies in index.html.

Voice layer (all browser-native, in index.html)

  • Speech-to-text via Web Speech API; spoken replies via speechSynthesis (TTS); an audio-reactive orb via Web Audio.
  • Hands-free conversation loop: speak → 1.5s silence auto-sends → AI replies aloud → auto re-listens. Tap the orb to send now; the ✕ exits to text.
  • Speaks replies only for voice messages (typed messages stay silent). Audio stops when the chat window is closed/minimized (WM onHide hook).
  • Mobile: skips the orb's getUserMedia analyser (a parallel mic stream breaks SpeechRecognition on mobile) and uses continuous=false.

Contact forms (Leave a Message / Schedule a Call / Collaboration)

All three POST to functions/api/message.js → Contact Form 7 (form ID 338) on cms.drissi.xyz → Flamingo (stores every submission) + WP Mail SMTP (emails).

  • Public CF7 endpoint /wp-json/contact-form-7/v1/contact-forms/338/feedback. The Worker must send Referer/Origin: https://khalildrissi.com headers or CF7's spam check rejects the datacenter-IP request.
  • Treats CF7 mail_sent and mail_failed as success (Flamingo stores regardless — check WP Admin → Flamingo → Inbound).
  • Email: WP Mail SMTP via mailbox portfolio@cms.drissi.xyz (host mail.cms.drissi.xyz:465 SSL); CF7 "To" = khalil@drissi.org.
  • Attachments (voice/video/file) upload to Cloudflare KV (MESSAGE_FILES), served via /api/file/{id}; the link is appended to the message body.
  • History: FormSubmit was dropped (rate-limited) and replaced by this CF7 setup.

Blog

Content lives in WordPress (cms.drissi.xyz) and is pulled by build.js at build time. 42+ posts (21 topics × EN/FR) with Polylang. The cli/ scripts create and manage posts via the WP REST API (auth Basic base64(user:app_password); Polylang lang param; featured media upload; category/tag handling).

Admin CMS panel

/admin/index.html edits the portfolio JSON and saves via /api/save-content → GitHub API → commit → deploy.

8Known issues & gotchas

  • UTF-8 / mojibake: save-content.js must decode base64→string with TextDecoder('utf-8') + Uint8Array, not atob() (which corrupts UTF-8). A stray 👋 emoji in content/en.json once corrupted into tofu rectangles — watch emoji in JSON content.
  • Ribbon / homepage data is edited via content/{lang}.json, never in index.html (build.js regenerates from JSON).
  • Emoji fonts must be LAST in the --mono / --sans font stacks — otherwise dual-presentation chars like ↗ (U+2197) render as colored emoji tiles instead of text arrows.
  • Image sources: Picsum and source.unsplash.com both return 503/405 — use LoremFlickr / Pexels.
  • CF7 spam check rejects requests without Referer/Origin headers (see §5).

9Pending work / TODO

Contrast & typography (from this session's audit — measured, not implemented)

  • P1 --fg-faint is rgba(255,255,255,0.30) ≈ 2.6:1 (fails WCAG AA) yet is used for real content (GitHub stats, usernames, repo langs). Split it: keep 0.30 for hairlines only, add a text-muted token ≈ 0.6 (~6:1).
  • P1 Accent #7c4dff as small text ≈ 3.5:1 (fails). Add a lighter violet (≈ #a78bfa) for accent text/links; keep #7c4dff for fills/shadows.

Claude Code hooks are wired in: the top-level .claude/settings.json has PreToolUse hooks that nudge Claude to run graphify query before grepping/reading source (only active when graphify-out/graph.json exists), and the top-level CLAUDE.md carries the graphify rules. Both are in the zip.
graphify install caveat (Mac): this machine's Python couldn't reach PyPI directly for a while (only curl/git/brew had egress). If pipx install times out on the new machine, it's likely a firewall (e.g., Little Snitch) blocking the python3 binary — allow it and retry. Also: graphifyy needs Python 3.10+ and pulls ~30 tree-sitter grammar packages; on very new Pythons prefer pipx install graphifyy --python "$(brew --prefix python@3.12)/bin/python3.12".

11Continuing with Claude Code on the new machine

  1. Install Claude Code and open the unzipped calil-drissi.dev/ folder.
  2. Claude Code auto-reads the top-level CLAUDE.md (graphify rules). It does not read this HTML automatically — paste it (or its key sections) into the first message so the new session has full context.
  3. Important: the assistant's persistent memory (project overview, credentials, blog content notes, known-issue fixes) lives in ~/.claude/…/memory/ on this machine and does not transfer. This HTML is that memory, serialized.
  4. Optional but recommended: install graphify (§10) so the codebase-query workflow works.
  5. Verify the toolchain: node build.js should produce dist/ cleanly before you start changing things.
